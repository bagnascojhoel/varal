## FEATURE ANALYSIS

**Story**: A Laundry Expert starts a drying session by selecting clothing weight
categories, creating a tracked session in SQLite. **Type**: New Feature —
Full-stack

**Functional Requirements**:

- :white_check_mark: Select one or more clothing weight categories (extra-light,
  light, medium, heavy, extra-heavy)
- :white_check_mark: Validate at least one category selected before enabling CTA
- :white_check_mark: Create one `DryingSession` record per selected category
  (batch insert). Each has startedAt=now, endedAt=null. Status is derived:
  endedAt=null → ACTIVE
- :white_check_mark: For each requested category, check if an active session for
  the same category was started within 5 minutes. If all categories conflict
  → 409. If some conflict → 206 (create the rest, report conflicts). If none
  → 201.
- :white_check_mark: Reject empty or invalid category values
- :white_check_mark: Return created session IDs to client; store in localStorage
- :white_check_mark: Transition UI to minimal active session view
- :white_check_mark: UI groups sessions started within 5 minutes together
  (collapsible)

**UI/UX Requirements Identified**:

- Category checklist with 44x44px touch targets
- "Start Session" CTA follows glass button pattern, disabled until >= 1 category
  selected
- Mobile-first layout under `/backoffice` route group
- Minimal active session view (category list, no timers — US-48 handles timers)
- Active sessions grouped by proximity (within 5 min) with collapse/expand

## 2. Context

### 2.1 Business Context & Problem Statement

**Problem**: The app currently only answers "should I wash today?" — it has no
session tracking. Users want to start a drying session, select which clothing
weight categories are on the line, and track them independently. This is the
foundation for the entire drying-session-tracker feature (US-46–52).

**User Journey**:

1. User navigates to `/backoffice`
2. Sees category selection checklist (5 weight categories)
3. Selects one or more categories → "Start Session" CTA enables
4. Clicks "Start Session" → POST to API creates N sessions (one per category)
5. Session IDs stored in localStorage; UI transitions to active session view
6. Active sessions started within 5 min are grouped together (collapsible)

**Acceptance Criteria**:

- [ ] POST endpoint creates one DryingSession per category (batch) in SQLite
- [ ] If all requested categories conflict (same category active within 5 min) →
      409, nothing created
- [ ] If some conflict but at least one is creatable → 206, created sessions +
      conflicting categories in response
- [ ] If no conflicts → 201, all created
- [ ] Validation error (400) for empty or unrecognised categories
- [ ] Session IDs persisted to localStorage on success
- [ ] UI transitions to a minimal active session view showing active sessions
- [ ] Active sessions started within 5 min are visually grouped (collapsible)
- [ ] Prisma + SQLite fully bootstrapped and wired into DI

## 3. Decision

### What Was Decided

**Prisma + SQLite as the persistence layer, with a `DryingSessionRepository`
port in the domain and a Prisma adapter in infrastructure.**

### Why This Approach Was Chosen

**Prisma + SQLite** (vs. JSON file storage / in-memory Map):

:white_check_mark: SQLite is zero-config, file-based — no external DB process
needed :white_check_mark: Prisma gives type-safe queries, migrations, and schema
evolution for upcoming US-47–52 :white_check_mark: Fits the existing Ports &
Adapters pattern — Prisma stays in infrastructure, domain stays pure
:white_check_mark: Auto-increment IDs are compact and efficient

### API Path Convention

- API endpoint: `POST /api/sessions` (domain-relevant, not nested under
  `/backoffice/`)
- UI route: `/backoffice` (groups all drying tracker pages)

### Domain Model Scope

Each `DryingSession` row maps 1:1 to a clothing weight category being dried.
Selecting multiple categories creates multiple independent rows sharing the same
`startedAt`. The Prisma schema includes `endedAt` for forward-compatibility
(US-51) but US-46 does not set it. The domain model is designed to be extended,
not rewritten.

### 4.2 Data Flow

```
[CategoryChecklist] → selects categories ["light", "heavy"]
        ↓
[StartSessionButton] → onClick
        ↓
POST /api/sessions { categories: ["light", "heavy"] }
        ↓
[route.ts] → Zod validation → container.get(DryingSessionApplicationService)
        ↓
[DryingSessionApplicationService.startSession(categories)]
  1. Fetch active sessions for the requested categories (WHERE endedAt IS NULL AND category IN (...))
  2. Partition requested categories into:
     - conflicting: active session for same category with startedAt within last 5 min
     - creatable: everything else
  3. If all conflict → return result with created=[], conflicting=[...]
  4. If partial or none conflict → create DryingSession entities for creatable categories, persist
  5. Return StartSessionResult { created: DryingSession[], conflicting: ClothingWeightCategory[] }
        ↓
[DryingSessionRepositoryPrismaAdapter]
  → prisma.dryingSession.createMany(...)  or  N × prisma.dryingSession.create(...)
        ↓
[route.ts] maps StartSessionResult to HTTP response:
  - created.length > 0 && conflicting.length === 0  → 201
  - created.length > 0 && conflicting.length > 0    → 206
  - created.length === 0                             → 409
        ↓
Response body (standard shape for all 3 cases):
  { created: [{ id, category, startedAt }], conflicting: ["light"] }
        ↓
[Client] → store created session IDs in localStorage → show ActiveSessionView
        ↓
[ActiveSessionView] → groups sessions by startedAt proximity (≤ 5 min) for display
```

### 4.3 Prisma Schema

```prisma
model DryingSession {
  id        Int      @id @default(autoincrement())
  category  String   // extra-light | light | medium | heavy | extra-heavy
  startedAt DateTime @default(now())
  endedAt   DateTime?
}
```

**Key design decisions**:

- Each `DryingSession` row represents one category being dried. Selecting 3
  categories creates 3 rows.
- `status` is **not** stored in the database. It is a derived property resolved
  in the `DryingSession` constructor: `endedAt == null` → ACTIVE,
  `endedAt != null` → COMPLETED.
- `endedAt` is included for forward-compatibility (US-51) but not set by US-46
  logic.

window are grouped together in the UI. This is a **presentation-only** concern —
it does not affect the domain model or database. The group heading shows the
earliest `startedAt` in the group.

## 5. Implementation Details

### 5.1 Domain Entities

**`ClothingWeightCategory`** — enum with 5 values. Validated at API boundary via
Zod, used in domain entities.

**`DryingSession`** — entity class (one instance = one category being dried):

- `id: number`
- `category: ClothingWeightCategory`
- `startedAt: Date`
- `endedAt: Date | null`
- `get status(): DryingSessionStatus` — derived getter: `endedAt == null` →
  `ACTIVE`, `endedAt != null` → `COMPLETED`
- `get isActive(): boolean` — convenience shorthand for `status === ACTIVE`
- Static factory:
  `DryingSession.start(category: ClothingWeightCategory): DryingSession` (sets
  `startedAt = now()`, `endedAt = null`)
- Static batch factory:
  `DryingSession.startMany(categories, existingActive): StartManyResult`
  Partitions requested categories against existing active sessions using the
  5-minute conflict rule (domain logic). Returns
  `{ created: DryingSession[], conflicting: ClothingWeightCategory[] }`. Created
  instances share the same `startedAt`.

### 5.2 Port Interface

```typescript
export interface DryingSessionRepository {
  createMany(sessions: DryingSession[]): Promise<DryingSession[]>;
  findActiveByCategories(
    categories: ClothingWeightCategory[],
  ): Promise<DryingSession[]>;
  // "active by categories" = WHERE endedAt IS NULL AND category IN (...)
}

export const DRYING_SESSION_REPOSITORY = Symbol.for('DryingSessionRepository');
```

### 5.3 Application Service

```typescript
export interface StartSessionResult {
  created: DryingSession[];
  conflicting: ClothingWeightCategory[];
}

@injectable()
export class DryingSessionApplicationService {
  constructor(
    @inject(DRYING_SESSION_REPOSITORY)
    private readonly repository: DryingSessionRepository,
  ) {}

  async startSession(
    categories: ClothingWeightCategory[],
  ): Promise<StartSessionResult> {
    const activeSessions =
      await this.repository.findActiveByCategories(categories);

    // Domain logic: partition categories and create non-conflicting sessions
    const { created, conflicting } = DryingSession.startMany(
      categories,
      activeSessions,
    );

    // Persist the creatable sessions (if any)
    const persisted =
      created.length > 0 ? await this.repository.createMany(created) : [];

    return { created: persisted, conflicting };
  }
}
```

### 5.4 API Response Convention

The route handler maps `StartSessionResult` to the appropriate HTTP status. The
response body uses a **standard shape** for all outcomes:

```typescript
// Response body (same shape for 201, 206, 409)
interface StartSessionResponse {
  created: { id: number; category: string; startedAt: string }[];
  conflicting: string[]; // category names that were skipped
}
```

| Condition                      | HTTP Status         | `created` | `conflicting` |
| ------------------------------ | ------------------- | --------- | ------------- |
| All categories created         | 201 Created         | N items   | `[]`          |
| Some created, some conflicting | 206 Partial Content | ≥1 items  | ≥1 items      |
| All categories conflict        | 409 Conflict        | `[]`      | N items       |

This is handled in the route handler, not via thrown errors — the application
service returns a `StartSessionResult` and the route maps it to the correct
status code.

### 5.5 PrismaClient DI Integration

A singleton `PrismaClient` is registered in `ContainerConfig.ts` as a constant
value (same pattern as `RestClientService` instances). Token:
`PRISMA_CLIENT = Symbol.for('PrismaClient')`.

### 5.5.1 Conflict Window Configuration

The 5-minute conflict window in `DryingSession.startMany` is driven by the
`SESSION_CONFLICT_WINDOW_MS` environment variable (defaults to `300000` — 5
minutes). E2E tests set it to `5000` (5 s) via `playwright.config.ts` so
scenario 5 can await 6 s instead of 6 minutes.

### 5.7 E2E API Tests (Playwright)

**Spec file**: `e2e/tests/sessions.spec.ts` **Client helper**:
`e2e/tests/helpers/sessions-client.ts`

A typed `SessionsApiClient` class (modelled after `RestClientService`) wraps all
`POST /api/sessions` calls and response parsing. Tests use it instead of raw
`request.*` calls. A `deleteAll()` method resets the SQLite table between tests
via a `DELETE /api/sessions` endpoint that is only active when
`TEST_HELPERS=true`.

The Playwright config sets `SESSION_CONFLICT_WINDOW_MS=5000` on the Next.js dev
server so scenario 5 can use a 6 s `waitForTimeout` instead of 6 minutes.

```typescript
// e2e/tests/helpers/sessions-client.ts
export class SessionsApiClient {
  constructor(private readonly request: APIRequestContext) {}

  async startSession(
    categories: string[],
  ): Promise<{ status: number; body: StartSessionResponse }> {
    const res = await this.request.post('/api/sessions', {
      data: { categories },
    });
    return { status: res.status(), body: await res.json() };
  }

  async deleteAll(): Promise<void> {
    await this.request.delete('/api/sessions');
  }
}
```

## 6. Consequences

### 6.1 Positive

:white_check_mark: Establishes the persistence foundation for all 7
drying-tracker stories :white_check_mark: Follows existing hexagonal patterns —
no architectural deviation :white_check_mark: SQLite requires zero
infrastructure setup :white_check_mark: Forward-compatible Prisma schema
minimises migration churn for US-47–52 :white_check_mark: Auto-increment IDs are
storage-efficient

### 6.2 Negative

:warning: Adds Prisma as a dependency (~2MB) and requires `prisma generate` in
the build pipeline :warning: SQLite file must be excluded from version control
(.gitignore) :warning: No auth on `/api/sessions` or `/backoffice` — acceptable
for now but must be addressed before production

### 6.3 Risks

| Risk                                                | Mitigation                                                               |
| --------------------------------------------------- | ------------------------------------------------------------------------ |
| Prisma Client not initialised in dev/prod correctly | Singleton pattern with lazy init; documented in README                   |
| SQLite file locking in serverless (Vercel)          | Currently dev-only; revisit DB choice if deploying to serverless         |
| Race condition on "check recent then create"        | Acceptable for single-user app; add unique partial index if needed later |
| `prisma generate` forgotten after schema changes    | Add to `postinstall` script in package.json                              |

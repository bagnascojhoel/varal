## 📋 FEATURE ANALYSIS

**Story**: Enforce passing builds before PR merges with automated testing, type checking, formatting validation, and static code analysis.
**Type**: Architecture Change — Infrastructure

**Functional Requirements**:

- ✅ Run TypeScript type checking on every pull request
- ✅ Run unit tests (Jest) on every pull request
- ✅ Run behavior tests (Cucumber) on every pull request
- ✅ Run code formatting checks (Prettier) on every pull request
- ✅ Run ESLint static analysis on every pull request (via `npm run check`)
- ✅ Run production build on every pull request
- ✅ Generate Prisma client before tests
- ✅ Block PR merge if any check fails
- ✅ Provide AI-powered code review feedback

**UI/UX Requirements Identified**: N/A — infrastructure only

## 2. Context

### 2.1 Business Context & Problem Statement

**Problem**: Without automated CI/CD, PRs can be merged with broken code, missing type safety, untested changes, or style violations. Manual review is slower and error-prone. The project lacks enforced quality gates.

**Current State**:
- No `.github/` directory
- No automated tests in the CI pipeline
- No static analysis (no ESLint config exists yet)
- Manual code review without automated assistance

**Acceptance Criteria**:

- [ ] GitHub Actions workflow runs on all pull requests to `main`
- [ ] All quality checks pass before PR merge is allowed
- [ ] AI code review comments appear automatically on pull requests
- [ ] ESLint runs in CI and reports violations without breaking local dev workflow
- [ ] Developers receive instant feedback on build failures

## 3. Decision

### What Was Decided

**GitHub Actions CI/CD with CodeRabbit (free AI review) + ESLint (CI-only static analysis)**

### Why This Approach Was Chosen

**GitHub Actions** (vs. CircleCI / GitLab CI / Jenkins):
- ✅ Native GitHub integration — no external account setup
- ✅ Free minutes for private repos (2,000 min/month on free plan)
- ✅ Simple YAML configuration in `.github/workflows/`
- ✅ Mature, well-documented ecosystem

**CodeRabbit** (vs. Anthropic's claude-code-action / GitHub's native review):
- ✅ Completely free for individual repos (public and private)
- ✅ GitHub App integration — no secrets management
- ✅ Automatically reviews all pull requests
- ✅ Posts inline comments with suggestions
- ✅ Zero configuration required (integrates via GitHub marketplace)

**ESLint** (vs. SonarCloud / CodeQL / Codacy):
- ✅ Completely free — no external service, no secrets, no account
- ✅ Runs entirely within GitHub Actions (no third-party dependency)
- ✅ Industry standard for TypeScript projects (`@typescript-eslint`)
- ✅ Available as `npm run lint`; not wired into `npm run build` or pre-commit hooks
- ✅ SonarCloud rejected: free plan requires public repository
- ✅ CodeQL rejected: requires GitHub Advanced Security (paid for private repos)

### CI Workflow Structure

**Single unified workflow** (`ci.yml`) triggered on `pull_request` to `main`:

```
1. npm ci + npx prisma generate
2. npm run check          (typecheck + format:check + lint)
3. npm test               (Jest unit tests)
4. npm run test:behavior  (Cucumber BDD)
5. npm run build          (Next.js production build)
6. cd e2e && npm ci + playwright install --with-deps chromium
7. npm run test:e2e       (Playwright E2E tests)
```

**Why a single workflow?**
- Cleaner dashboard (one status check per commit)
- All checks share the same environment (DRY)
- Faster feedback when early checks fail (fail fast)

### Branch Protection

GitHub branch protection rules (configured in GitHub UI, not files) will:
- Require CI workflow status checks to pass
- Dismiss stale pull request approvals when new commits are pushed

**Decision**: Document in ADR; configure in GitHub → Settings → Branches → main → Require status checks to pass before merging.

### AI Code Review Strategy

**Primary**: CodeRabbit (free, zero-config)
- Installed via GitHub Marketplace
- Automatically reviews all PRs
- No API keys or secrets needed

**Optional Secondary**: Anthropic claude-code-action (requires `ANTHROPIC_API_KEY` secret)
- Free API tier available
- Can be added later if more detailed reviews are desired

## 4. Architectural Design

### 4.1 GitHub Actions Workflow Architecture

```
┌─────────────────────────────────────────────────────────────┐
│  Pull Request Opened/Updated on main                         │
└──────────────────┬──────────────────────────────────────────┘
                   │
        ┌──────────▼──────────┐
        │  Checkout Code      │
        │  npm ci             │
        │  prisma generate    │
        └──────────┬──────────┘
                   │
               ┌──────────────┬──────────────┐
               │              │              │
               ▼              ▼              ▼
        ┌───────────┐  ┌──────────┐  ┌────────────┐
        │   check   │  │   test   │  │    build   │
        │(tsc+fmt+  │  │  (Jest)  │  │  (Next.js) │
        │  eslint)  │  └──────────┘  └────────────┘
        └─────┬─────┘
              │
                   ┌──────────▼──────────┐
                   │  test:behavior      │
                   │  (Cucumber)         │
                   └──────────┬──────────┘
                              │
                   ┌──────────▼──────────┐
                   │  npm run build      │
                   │  (Next.js)          │
                   └──────────┬──────────┘
                              │
                   ┌──────────▼──────────┐
                   │  npm run test:e2e   │
                   │  (Playwright E2E)   │
                   └──────────┬──────────┘
                              │
            ┌─────────────────▼─────────────────┐
            │ CodeRabbit AI Review               │
            │ (GitHub App, runs automatically)   │
            └───────────────────────────────────┘
```

### 4.2 Data Flow

```
Developer pushes commit to PR
    ↓
GitHub triggers ci.yml workflow
    ↓
Node 24 environment starts:
  - npm ci (lock-file based install)
  - prisma generate (generates Prisma client)
    ↓
Checks run in parallel/sequence:
  - check: tsc --noEmit && prettier --check . && eslint src
  - test: jest (unit tests)
  - test:behavior: cucumber (BDD)
  - build: next build (production bundle)
  - test:e2e: playwright (E2E — Mockoon + Next.js dev server)
    ↓
If all pass:
  - CodeRabbit reviews code, posts inline comments
  - Branch protection allows merge
    ↓
If any fail:
  - CI status = ❌ on PR
  - Developer sees logs, fixes issues
  - Re-push updates PR (workflow re-runs)
```

## 5. Implementation Details

### 5.1 File Structure

```
.github/
  workflows/
    ci.yml                         NEW — Main CI workflow

.eslintrc.json (or eslint.config.js) NEW — ESLint config (CI picks it up automatically)

.ai/features/github-actions-ci/
  ADR-2026-03-04.md               THIS document
```

> Note: `sonar-project.properties` is **not** created — SonarCloud was dropped in favour of ESLint.

### 5.2 Environment & Dependencies

**Runtime Requirements**:
- Node.js 24+ (GitHub Actions runner)
- npm 10+ (for `npm ci`)
- SQLite (for Prisma; uses `DATABASE_URL` env var)
- Prisma CLI (installed via `npm ci`)

**New dev dependencies** (added to `package.json` devDependencies):
- `eslint`
- `@typescript-eslint/parser`
- `@typescript-eslint/eslint-plugin`

**New npm script** (added to `package.json` scripts):
- `check`: runs `tsc --noEmit && prettier --check . && eslint src` — combines typecheck, format check, and lint

**Secrets Required** (in GitHub repo settings):
- `DATABASE_URL` — SQLite database path (for Prisma during build step)

**No secrets needed** for CodeRabbit or ESLint.

### 5.3 ESLint Configuration

Minimal `@typescript-eslint` config targeting `src/`:

```json
{
  "parser": "@typescript-eslint/parser",
  "plugins": ["@typescript-eslint"],
  "extends": [
    "eslint:recommended",
    "plugin:@typescript-eslint/recommended"
  ],
  "root": true
}
```

ESLint is invoked via `npm run check`, which combines typecheck, format:check, and lint into a single script. It is **not** wired into `npm run build` or `npm run dev`.

## 6. Consequences

### 6.1 Positive

✅ **Enforced Quality Gates**: No broken code reaches `main`
✅ **Developer Feedback Loop**: Instant CI status on each commit
✅ **Reduced Manual Review**: AI reviews catch common issues (styling, logic)
✅ **Zero External Dependencies**: ESLint runs fully within GitHub Actions — no third-party accounts or secrets
✅ **Frictionless Local Dev**: ESLint available via `npm run lint` but not enforced on build
✅ **Free Infrastructure**: GitHub Actions (2k min/month free), CodeRabbit (free), ESLint (free)

### 6.2 Negative

⚠️ **Build Time**: CI pipeline adds ~1–2 minutes per commit
⚠️ **No Quality Dashboard**: Unlike SonarCloud, ESLint has no persistent metrics dashboard — issues are visible only in CI logs and PR annotations
⚠️ **Initial ESLint Violations**: First run may surface many existing violations; may need a grace period or baseline suppression
⚠️ **Learning Curve**: Team must understand GitHub Actions YAML syntax

### 6.3 Risks

| Risk | Mitigation |
|------|------------|
| First ESLint run blocks all PRs due to existing violations | Add `--max-warnings` flag or `// eslint-disable` comments as a short-term baseline; fix violations incrementally |
| CI takes too long, blocking development | Parallelize typecheck, format, eslint, and test jobs |
| Flaky tests cause false CI failures | Fix root cause of flaky tests; don't disable checks |
| Database setup fails in CI | Use SQLite file path in `DATABASE_URL` secret; run `prisma generate` (not `migrate`) in CI |
